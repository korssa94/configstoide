# 🏭 PLC Code Generator Pro Max

Автоматизированный инструмент для генерации кода ПЛК (программных блоков, GVL, ST) на основе конфигураторов ввода/вывода (ТЭ5), карт адресации и конфигураторов сигнализаций. Проект снабжён удобным веб-интерфейсом на базе Streamlit и полностью изолированным окружением.

---

## 🚀 Быстрый старт (Инструкция по развертыванию)

Инструмент спроектирован так, чтобы его установка и запуск на любом компьютере инженера выполнялись буквально «в два клика» без необходимости ручной настройки системных путей и глобальных библиотек Python.

### Шаг 1. Первая установка (Один раз)
1. Склонируйте репозиторий проекта на свой локальный компьютер.
2. В корневой папке проекта найдите и запустите файл **`install.bat`**.

> **Что произойдёт в этот момент:**
> * Скрипт проверит наличие Python в Windows. Если Python не установлен, скрипт самостоятельно скачает и установит официальный **Python 3.11** через системный менеджер пакетов `winget`.
> * *Если Python устанавливался с нуля, скрипт попросит вас закрыть консоль и запустить `install.bat` повторно для обновления системных переменных.*
> * В корне проекта автоматически создастся изолированное виртуальное окружение в папке `.venv`.
> * Внутри окружения обновятся все компоненты и установятся строго необходимые библиотеки (Streamlit, openpyxl, pandas и др.).

### Шаг 2. Запуск приложения (Повседневный)
Для повседневного запуска генератора используйте файл **`invisible_start.vbs`** в корне проекта.

> **Как это работает:**
> * Сервер Streamlit запускается в скрытом (фоновом) режиме, не раздражая вас постоянно открытыми чёрными окнами командной строки на панели задач.
> * Скрипт выждет 5–7 секунд, пока сервер инициализируется, и автоматически откроет вкладку генератора в вашем браузере по умолчанию по адресу: `http://localhost:8501`.

### Шаг 3. Остановка сервера (При необходимости)
Если вам необходимо полностью остановить фоновый сервер и высвободить оперативную память, перейдите в папку `.services/` и запустите файл **`stop_server.bat`**. Он принудительно и безопасно завершит процессы Python, привязанные к генератору.

## 📖 Глоссарий

Несколько терминов, которыми оперирует весь проект.

| Термин | Что означает |
|---|---|
| **Конфигуратор** | Excel-файл (.xlsm), описывающий сигналы для одного компонента системы |
| **ТЭ5** | Конфигуратор входов/выходов (аналоговых, дискретных, шлейфов). Файлы вида `*ТЭ5*.xlsm` |
| **ТБ51 (Д51, В8)** | Конфигуратор сигнализаций. Файлы вида `*Д51*.xlsm`, `*В8*.xlsm`, `*ТБ51*.xlsm` |
| **Мастер-конфигуратор** | Главный файл проекта с картой связей: какой ПЛК с каким ТЭ5 и каким ТБ51 работает. Файл `Конфигурация проекта*.xlsx` |
| **ПЛК** | Программируемый логический контроллер (один проект может работать на нескольких ПЛК) |
| **Подсистема** | Логический раздел системы (например, "Блок двигателя" или "Система вентиляции"). Описывается в листе "Настройки" каждого конфигуратора |

**Типы сигнализаций (значение колонки "Тип" в ТБ51):**

| Код | Означает | Куда попадает в коде |
|---|---|---|
| **ПС** | Предупредительная сигнализация | `alr.gvl` (если действие критическое из набора `ХР`/`ГР`/`БЗ`— ещё и `ppu.gvl`) |
| **ППУ** | Предпусковые условия | `ppu.gvl` |
| **АС** | Аварийная сигнализация | `trs.gvl` (тревожная), либо `crs.gvl` (критическая — если действие из набора `АОсс`/`АОбс`/`ВОсс`/`ВОбс`/`АО`/`ВО`/`Пожар`) |
| **ОС** | Ограничительная сигнализация | `lmt.gvl` |

## 🗂 Структура папки Configs

Все конфигураторы одного проекта лежат рядом в одной папке (например, `/Project/Configs/`). Минимальный комплект:

```
Configs/
├── Конфигурация проекта.xlsm          ← Мастер
├── GPA_SAU/                            ← Папка типа абонента
│   ├── ЛСНР.421457.001-ХХХ ТЭ5 vХХ.ХХ.ХХ.xlsm   ← Конфигуратор ТЭ5
│   └── ЛСНР.421457.001-ХХХ ТБ51 vХХ.ХХ.ХХ.xlsm  ← Конфигуратор ТБ51
```


### Связи через Мастер-конфигуратор

В Мастере на первом листе есть несколько ключевых строк:
- **"Контроллер"** — имена ПЛК (`GPA1_SAU`, `GPA2_SAU`, ...)
- **"Входы/выходы"** — какой файл ТЭ5 связан с этим ПЛК
- **"Сигнализации"** — какой файл ТБ51 связан с этим ПЛК

Структура колонок: каждая колонка = один ПЛК. По этой таблице приложение узнаёт, для каких ПЛК генерировать код для выбранного конфигуратора.

## 🖥 Интерфейс

### Путь к папке Configs
Поле ввода — путь к папке с конфигураторами. Программа выполняет поиск вниз от указанного пути. Можно указать как корень Configs, так и Configs/GPA_SAU, например. Запоминается между запусками (хранится в `user_settings.json`).

### Конфигураторы
Multiselect со списком всех найденных ТЭ5/ТБ51-файлов в указанной папке (и в подпапках). Можно выбрать один или несколько.

### Чекбоксы настроек процесса

| Чекбокс | Что делает | Применим к |
|---|---|---|
| **Создать исходники** | Генерирует `.gvl` и `.st` файлы в подпапку `source/{ПЛК}/` | ТЭ5, ТБ51 |
| **Обновить цвета ячеек** | Красит резервные строки серым (`COL_RESERVE`) | ТЭ5 |
| **Обновить текстовое описание** | (1) Заполняет колонки `Условие (текст)`, `Неисправность (текст)`, `Взвод (текст)`, `Сброс (текст)` расшифровкой кода через PLCopen XML-кэш. (2) Заполняет колонку `Условие` автоформируемых строк (`DI` / `Нет DI` / `< 20 °С` / `> 80 °С`) — данные из ТЭ5 | ТБ51 |
| **Обновить документ** | Заполняет лист `Документ` собранными данными и форматирует через VBA-макрос `Format_Document_Simple` из надстройки | ТБ51 |

При выборе нескольких конфигураторов разных типов (например, если сразу выбрать и ТЭ5, и ТБ51) можно проставлять все галочки. Программа будет работать по правилу "Применим к", т.е. галочка "Обновить текстовое описание", например, в любом случае не запустит функцию обновления, когда будет обрабатываться ТЭ5.

### PLCopen XML
Если включён "Обновить текстовое описание", появляется selectbox для выбора файла `*.plcopen.xml` (экспорт проекта CODESYS из папки `source.export/`). Из него строится кэш переменных для перевода кода в текст. Если файлов нет — соответствующий шаг пропускается без ошибки.

### Кнопка "Запустить процесс"
Стартует обработку. При обнаружении ошибок кросс-проверки процесс **останавливается** и появляется диалог с двумя кнопками:
- **"Всё равно создать (без ошибочных)"** — генерация в force-режиме: дефектные строки исключаются с WARNING в логе, остальные обрабатываются
- **"Отменить генерацию"** — полный откат, ничего не записывается

## ✅ Проверки

### ТЭ5: валидации внутри листа

Для каждой строки сигнала (где "Создавать код" = 1):
- **Пустое алгоритмическое имя** — поле обязательно
- **Пробелы в алгоритмическом имени** — недопустимы
- **Кириллица в алгоритмическом имени** — недопустима
- **Дубликаты адресов** — `Шкаф-Крейт-Модуль-Канал` должны быть уникальны в проекте
- **Отсутствие пределов измерения (MIN/MAX)** — для аналоговых сигналов (Taipar)
- **Порядок уставок** — для аналоговых должно выполняться: `MIN < LL < L1 < L < H < H1 < HH < MAX`
- **Наличие подсистемы** — должна совпадать с одной из объявленных в листе "Настройки"

### ТЭ5: перекрестные проверки между листами

Если в строке "Контроль цепи" имеет значение `1`, `2` или `3`, должен существовать парный контрольный параметр на другом листе:

| Лист | Условие | Должен существовать |
|---|---|---|
| `Вых.А сигн.` (Taopar) | Контроль цепи = `1` | `kcdi_{alg_name}` на листе `Вх.Д сигн.` |
| `Вых.А сигн.` (Taopar) | Контроль цепи = `2` | `kcao_{alg_name}` на листе `Вх.А сигн.` |
| `Вх.Д сигн.` (Tdipar) | Контроль цепи = `3` | `kcdi_{alg_name}` на том же листе |
| `Вых.Д сигн.` (Tdopar) | Контроль цепи = `1` | `kcdo_{alg_name}` на листе `Вх.Д сигн.` |
| `Вых.Д сигн.` (Tdopar) | Контроль цепи = `2` | `kcdo_{alg_name}` на листе `Вх.А сигн.` |

### ТБ51 → ТЭ5 (кросс-проверка между конфигураторами)

Только для строк, где **"Условие (код)" пустое** (то есть код собирается автоматически). Парсер обращается через Мастер к связанным ТЭ5 и проверяет:

- **"Уставка" пустая + "Условие" = "DI"** или **"Уставка" = "N"** → параметр должен существовать в ТЭ5 как `tdipar`
- **"Уставка" в [LL, L1, L, H, H1, HH]** → параметр должен существовать как `taipar` И соответствующая уставка должна быть **активна** (непустая ячейка в колонке "Порог АН" / "Порог Н" / ...)

При расхождении пишется ERROR с указанием конкретного ПЛК и файла ТЭ5, в котором проблема.

### Логика "Условие (код)"

Этот столбец в ТБ51 — переключатель режима работы парсера для каждой строки сигнализации:

| Состояние | Что делает парсер |
|---|---|
| **Пусто** | Код условия собирается автоматически из `Алг.имя сигнала` + `Уставка`, выполняется кросс-проверка с ТЭ5, при "Обновить текстовое описание" автоматически заполняется столбец "Условие" |
| **Не пусто** | Код берётся как есть (ручная конфигурация), кросс-проверка не выполняется, столбец "Условие" не перезаписывается |

То же относится к парным столбцам `Неисправность (код)`, `Взвод (код)`, `Сброс (код)`.

## 📤 Что генерируется

Для каждого ПЛК, привязанного к обрабатываемому конфигуратору, создаётся подпапка в `source/`:

```
{Configs_root}/
└── source/
    ├── GPA1_SAU/
    │   ├── 00_Глобальные переменные/
    │   │   └── subsystem.gvl               ← Список подсистем
    │   ├── 04_Входы_выходы/                ← Создаётся при обработке ТЭ5
    │   │   ├── aipar.gvl / aipar_update.st  (аналоговые входы)
    │   │   ├── aopar.gvl / aopar_update.st  (аналоговые выходы)
    │   │   ├── dipar.gvl / dipar_update.st  (дискретные входы)
    │   │   ├── dopar.gvl / dopar_update.st  (дискретные выходы)
    │   │   ├── fpl.gvl / fpl_update.st      (пожарные шлейфы)
    │   │   └── reset_io_var_stat.st
    │   └── 05_Сигнализации/               ← Создаётся при обработке ТБ51
    │       ├── alr.gvl  (предупредительные)
    │       ├── trs.gvl  (тревожные)
    │       ├── lmt.gvl  (ограничительные)
    │       ├── crs.gvl  (критические аварийные)
    │       ├── ppu.gvl  (предпусковые условия)
    │       ├── alr_update.st  (общий ST для alr/trs/lmt)
    │       ├── crs_update.st
    │       ├── ppu_update.st
    │       └── reset_alarm_var_stat.st
    ├── GPA2_SAU/
    └── GPA3_SAU/
```

- **`*.gvl`** — Global Variable List, декларации сигналов и инициализация
- **`*_update.st`** — Structured Text функция, в каждом цикле обновляющая состояние своей группы сигналов
- **`reset_*.st`** — сброс `VAR_STAT`

Кодировка генерируемых файлов — **UTF-16 LE с BOM** (настройка в `AppConfig.FILE_ENCODING`).

## 📎 Excel-надстройка

Файл: **`Надстройка для Excel/Надстройка.xlam`**

Содержит VBA-макрос **`Format_Document_Simple`** — форматирует лист "Документ" в конфигураторе ТБ51 после того, как Python заполнил его данными: объединяет ячейки, выставляет шрифты и ширины колонок, проставляет границы.

Используется только при включённом чекбоксе **"Обновить документ"**:
1. Python через xlwings/COM открывает конфигуратор
2. Заполняет лист "Документ" собранными данными
3. Активирует целевую книгу (важно — макрос работает с `ActiveWorkbook`)
4. Вызывает `Format_Document_Simple` через `app.api.Run(...)`
5. Сохраняет и возвращает фокус на исходную книгу

### Установка надстройки в Excel

Разработчик → Надстройки Excel → Обзор → выбрать `Надстройка для Excel/Надстройка.xlam` → ОК.

Если надстройка ранее уже устанавливалась и физически переехала в другую папку — Excel предложит обновить путь. Соглашайтесь.

После установки макрос будет доступен из любого открытого Excel-файла (но удобно использовать его именно из приложения).

### Поток управления одного прогона

```
app.py (кнопка нажата)
→ application.processor.process_configurator(filepath, …)
  → {TE5|Alarm}Parser.parse()
    ├── shared.parsers.base_parser  (column mapping, generate_header, и т.д.)
    ├── alarm.cross_validation      (только для AlarmParser — кросс с ТЭ5)
    └── модели в *_configurator/models  (Talr / Taipar / ...)
  → (если do_translation) alarm.documentation.text_updater  ─┐
  → (если do_document)    alarm.documentation.document_updater├── через
  → (если do_coloring)    shared.documentation.color_updater ─┘ seamless_excel
→ app.py пишет files_to_write на диск
```

### Ключевые соглашения

1. **Доменные префиксы в коде**: `inout` для входов/выходов, `alarm` для сигнализаций. Не используются продуктовые имена (`TE5`/`TB51`) в новых функциях/переменных — на случай ребрендинга. Существующий `TE5Config` — историческое исключение, переименование запланировано.

2. **Кодировка генерируемых файлов**: UTF-16 LE с BOM (через `AppConfig.FILE_ENCODING`).

3. **Логирование**: парсеры и апдейтеры принимают `logger` параметром (обычно `add_log`). Streamlit-зависимости изолированы в `application/logger.py` и в `cross_validation.py` (через `st.session_state.failed_rows_cache`).

4. **Бесшовная работа с Excel**: контекст-менеджер `seamless_excel` в `shared/documentation/excel_utils.py` — единая точка для всех апдейтеров. Запоминает активную книгу, замораживает экран, делает работу, восстанавливает фокус. Параметр `print_area_mode` (`"wipe"` / `"fix"` / `None`) выбирает стратегию работы с областями печати.

5. **Кэш ошибок кросс-проверки**: после первого прогона `failed_rows_cache` хранится в `st.session_state`. При нажатии "Всё равно создать" парсер использует кэш для исключения дефектных строк без повторного чтения ТЭ5.

## 🆘 Troubleshooting / FAQ

### Макрос форматирования не выполнился
**Симптом**: в логе появилось `❌ Не удалось выполнить макрос Format_Document_Simple`.
**Причина**: надстройка `Надстройка.xlam` не зарегистрирована в Excel или путь к ней устарел.
**Решение**: Файл → Параметры → Надстройки → Перейти → подключите `Надстройка.xlam`. Перезапустите Excel и приложение.

### Перевод текстов не работает: "Файлы .plcopen.xml не найдены"
**Симптом**: WARNING в логе при включённом "Обновить текстовое описание".
**Причина**: в дереве проекта нет файла `*.plcopen.xml` в папках `source.export/`.
**Решение**: экспортируйте проект из CODESYS в формат PLCopen. Без XML переводы трёх *.текст-колонок пропускаются, но столбец "Условие" по-прежнему заполняется (он использует данные из ТЭ5, не из XML).

### Permission denied при записи `.gvl`/`.st`
**Симптом**: ERROR в логе вида `Не удалось записать '...': PermissionError`.
**Причина**: файл открыт в редакторе (VSCode, CODESYS) и заблокирован.
**Решение**: закройте файл в редакторе, перезапустите процесс. Остальные файлы при этом успешно записываются — приложение не валится из-за одного заблокированного.

### Excel завис или показывает "восстановление документа"
**Причина**: одновременная попытка нескольких процессов работать с файлом, или незакрытый процесс Excel в фоне.
**Решение**: остановите все Excel-процессы через диспетчер задач, перезапустите приложение через `invisible_start.vbs`.

### Расхождение уставок между ПЛК
**Симптом**: WARNING вида `Строка N: расхождение уставки 'LL' параметра '...' между ПЛК (GPA1_SAU: 20 °С, GPA2_SAU: 25 °С) — столбец 'Условие' не обновлён`.
**Причина**: для одной алгоритмической переменной в разных ТЭ5 заданы разные значения порога.
**Решение**: либо это намеренная разница (тогда столбец "Условие" в ТБ51 заполните вручную, поставив "Условие (код)" непустым — приложение перестанет его трогать), либо ошибка конфигурации — выровняйте значения в ТЭ5.

### Уставка помечена "изменяемая" в ТЭ5
**Симптом**: WARNING `Строка N: уставка 'LL' помечена 'изменяемая' — столбец 'Условие' не обновлён`.
**Причина**: в ТЭ5 значение порога — не число, а пометка "изменяемая" (задаётся оператором на ходу).
**Решение**: ожидаемое поведение, не баг. Заполните "Условие" в ТБ51 вручную либо оставьте как есть.

### "Анализ" в логе дублируется
**Причина**: в multiselect случайно попали два разных пути с одинаковым именем файла (например, копия в подпапке `Archive/`). 
**Решение**: с недавних пор есть защитный дедуп по basename — второй файл пропускается с WARNING. Проверьте дерево `/Configs/` на дубликаты в подпапках.

### Области печати "съехали" после прогона ТЭ5
**Причина**: ранее существовавший баг — `repair_and_cleanup_file` выжигал все Print_Area имена.
**Решение**: исправлено через `print_area_mode="fix"` в `seamless_excel`. Для ТЭ5 теперь работает аккуратное переименование `Print_Area` → `Область_печати` без потери координат. В новых версиях не должно повторяться.
